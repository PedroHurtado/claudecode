La regla de oro

Revisa el diff antes de comitear

Claude acierta la mayoría de las veces, y «la mayoría» no es «siempre». Usa el commit como checkpoint: revisa con git diff o /diff (un visor interactivo de todos los cambios) y solo entonces comitea.

Se cierra un círculo de la Sesión 2

El comando /commit, con su contexto completo

!`git diff --cached`

Commits) a partir del diff de arriba.

Un commit bien hecho parte del diff real, no de lo que el modelo cree que cambió.

Pull requests

Genera el PR completo — pero lo validas tú

git push -u origin feature/pagos

gh pr create \
  --title "Integración de pagos" \
  --base main
Criterio, no automatismo

Otra persona va a revisar ese PR. La descripción tiene que reflejar de verdad qué cambia y por qué. Es un borrador muy bueno que tú validas.

Resolución de conflictos

Claude propone, tú revisas qué se quedó de cada lado

Entiende el contexto de ambos lados y propone la fusión. Pero un conflicto mal resuelto mezcla de forma sutil dos intenciones: la cautela es mayor que en un commit normal.

La mejor resolución de conflictos es prevenirlos.

→ Y eso nos lleva directos a los worktrees.

El problema que resuelven los worktrees

Entra un bug urgente y tus opciones son todas malas

Un git stash pierde contexto; clonar de nuevo cuesta diez minutos. Y dos sesiones en el mismo directorio se pisan los cambios y producen salida corrupta. Clonar el repo por cada tarea (varios gigas) no escala.

El concepto

Qué es un git worktree

mi-proyecto/          ← principal (main)
mi-proyecto-pagos/    ← feature/pagos
mi-proyecto-refactor/ ← feature/refactor

Directorio separado, su propia rama, comparte historia y remoto. Independiente en disco, sin duplicar la historia.

Las ediciones de una sesión nunca tocan los ficheros de otra.

Soporte nativo

Cómo se usan en Claude Code

# Terminal 1
claude -w feature-pagos

# Terminal 2 · en paralelo
claude -w bugfix-auth

Crea un worktree aislado bajo .claude/worktrees/, una rama nueva, y una sesión acotada a ese directorio. Tu árbol principal queda intacto.

Mientras Claude trabaja en uno, tú revisas otro: no esperas, diriges.

Detalles que ahorran disgustos

Cuatro cosas que conviene saber

Ignóralos en git

Añade .claude/worktrees/ al .gitignore.

Parten de la rama base limpia

Si A depende de B, no las corras en paralelo: construye B, mergéala, empieza A.

Limpieza

Los -p no se limpian solos: git worktree remove.

Config de entorno

Copia .env y puertos con un .worktreeinclude.

Un puente con la Sesión 3

Los subagentes también pueden aislarse en worktrees

isolation: worktree

Añádelo al frontmatter de un subagente y cada uno recibe un worktree temporal que se elimina solo al terminar sin cambios. Es la combinación que permite paralelismo real sin colisiones.

Paralelizar a lo loco es un antipatrón

Cuándo ayuda y cuándo estorba

✓Brillan cuando
  • Hay features independientes a la vez
  • Sigues tu trabajo mientras Claude corre una tarea larga
  • Atiendes un bug urgente sin destruir tu contexto
✕Estorban cuando
  • Las tareas dependen entre sí (rama base limpia)
  • Son tan pequeñas que el overhead de montar y limpiar no compensa
Para practicar

Tres tareas

1Dos worktrees con claude -w, tareas independientes; comprueba que no se pisan.
2Lleva una feature al PR (git push, gh pr create); revisa con /diff y valida la descripción.
3Provoca una dependencia entre worktrees; observa el aislamiento y resuélvelo con el orden correcto.
