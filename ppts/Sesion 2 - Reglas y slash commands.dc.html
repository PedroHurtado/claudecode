Sesión 2 · Contexto, reglas y comandos
Dónde estamos

Del CLAUDE.md a hacer que Claude actúe

Ya tienes memoria persistente del proyecto. Pero el CLAUDE.md tiene dos límites que hoy resolvemos.

1
Se carga entero, siempre

No escala si lo metes todo: ocupa contexto incluso en sesiones donde no pinta nada.

→ Reglas con ámbito

2
Dice qué saber, no qué hacer

Describe contexto, no un procedimiento paso a paso para los flujos que repites.

→ Slash commands

El problema que resuelven las reglas

Por qué el CLAUDE.md raíz no escala solo

Crece sin freno

A partir de ~200 líneas se comporta como un documento largo: difícil de mantener y fácil de contradecir.

Carga lo que no toca

Hay instrucciones que solo aplican a una parte del código.

«Los componentes de React llevan tests con Testing Library» no pinta nada cuando tocas el backend, pero ocupa contexto igual.

La solución es la misma para los dos: modularizar.

La solución

El raíz se queda como índice; el detalle vive en reglas con ámbito

El CLAUDE.md raíz, pequeño y estable. Todo lo profundo, específico de una zona o solo relevante a veces, se mueve a ficheros bajo .claude/rules/.

Dos vías para conectarlas

Cómo enchufas una regla al contexto

Por referencia @

El CLAUDE.md actúa de índice y el detalle vive aparte, cargándose cuando hace falta.

@docs/api-standards.md
Con ámbito de ruta

Las instrucciones de una carpeta entran en juego solo cuando trabajas ahí, no siempre.

.claude/rules/ · CLAUDE.md por subdirectorio

En un monorepo: lo común arriba, lo específico cerca del código al que aplica.

Regla de oro para decidir dónde va una instrucción
Un revisor levantaría una ceja

Es un criterio que se interpreta, no una garantía dura.

→ CLAUDE.md o regla
Bloquearía un merge en CI

Es una regla dura que no puede depender de interpretación.

→ CI o un hook

El CLAUDE.md y las reglas no sustituyen a la automatización; la complementan.

Para que se cumpla

Los mismos principios que el CLAUDE.md

01
Imperativo, no observación

«Nunca uses any sin un comentario que lo justifique», no «intentamos evitar any».

02
Concreta y verificable

Una regla que no puedes comprobar si se cumple está mal escrita.

03
Énfasis con moderación

IMPORTANTE / DEBES solo en lo verdaderamente crítico. Si todo es importante, nada lo es.

Un ejemplo real

Una regla con ámbito para los tests de frontend

# Reglas de testing de componentes

- Usa React Testing Library, nunca Enzyme.
- Testea comportamiento visible para el usuario,
  no detalles de implementación.
- Cada componente nuevo en src/components/
  necesita su *.test.tsx.
- Los datos de prueba salen de factorías,
  nunca mocks en línea.

Con ámbito sobre src/components/, esto no gasta contexto cuando trabajas en el backend.

Todo en imperativo y verificable: cada línea se puede comprobar.

Segunda pieza

Slash commands

Empaqueta en un fichero el prompt que repites cada día y lánzalo tecleando / y su nombre. Lo escribes una vez; lo usas cincuenta.

El mecanismo

Un fichero markdown = un comando

El nombre del fichero es el comando; su contenido, el prompt que recibe Claude al invocarlo.

.claude/commands/
review.md
«Revisa este diff buscando bugs,
organiza en crítico/importante/menor…»
→
/review

Invocable en tu sesión, una y otra vez.

Dos ámbitos

Proyecto vs. personal

Ámbito
Ubicación
Para qué
Proyecto
.claude/commands/
Comandos del equipo. Se versionan con git: quien clona los hereda.
Personal
~/.claude/commands/
Tus comandos, en todos tus proyectos, en tu máquina. No se versionan.
2026

Comandos y skills se fusionaron (v2.1.101). Los ficheros en .claude/commands/ siguen funcionando —es la vía más rápida para empezar—, pero la doc los marca como legacy y recomienda el formato de skill para lo nuevo. Las skills son el tema de la Sesión 3.

Anatomía · ingrediente 1

Frontmatter: metadatos y restricciones

---
description: Revisión de seguridad
allowed-tools: Read, Grep, Glob
---
Analiza el código en busca de
vulnerabilidades: inyección SQL,
XSS, credenciales expuestas…
description

El texto que aparece al navegar los comandos con /. Hace tu comando descubrible.

allowed-tools

Restringe qué tools usa Claude. Solo Read, Grep, Glob = puede leer pero no modificar nada. Clave por seguridad.

Anatomía · ingrediente 2

Argumentos: pasa el objetivo al invocar

$ARGUMENTS

Captura todo lo que escribes tras el nombre, como un único string.

/review-colors grafico.py
→ Revisa el fichero grafico.py
$0 · $1 · …

Posicionales por índice, base cero: $0 es el primero. Para cuando esperas varias piezas.

Arregla el issue #$0
con prioridad $1
Anatomía · ingrediente 3

Salida de shell embebida con !

---
description: Crea un commit convencional
allowed-tools: Bash(git diff:*),
  Bash(git commit:*)
disable-model-invocation: true
---
!`git diff --cached`

Redacta el mensaje a partir del
diff de arriba (Conventional Commits).

El ! ejecuta el comando antes y mete su salida en el prompt. Claude redacta con el diff real delante.

Dos guardarraíles

Pre-aprueba el shell en allowed-tools para que no pida permiso. Y disable-model-invocation: true para que un comando con efectos solo se dispare cuando lo tecleas tú.

El temario, construido

Cuatro comandos reales

/explain

Explica un fichero o función a quien entra nuevo: qué hace, cómo encaja, las decisiones no obvias.

allowed-tools: Read, Grep, Glob
/fix

Localiza la causa raíz leyendo antes de proponer, aplica el arreglo mínimo y corre los tests.

…Read, Grep, Glob, Edit, Bash(make test:*)
/test

Detecta el framework ya en uso, cubre camino feliz, límites y errores, y verifica que pasan.

…Read, Grep, Glob, Edit, Bash(make test:*)
Crear comandos

El meta-paso: crear un comando es crear un .md en la carpeta. Nada que instalar. El propio Claude puede escribírtelo.

Descubrir y probar

Sesión 3
Errores comunes

Qué falla y cómo se diagnostica

Síntoma
Causa y arreglo
«Modifica cosas que no debería»
Falta allowed-tools. Un comando de lectura lleva solo Read, Grep, Glob.
«$ARGUMENTS no recoge lo que paso»
Confusión con los posicionales. ¿Todo como string, o piezas separadas?
«Me pide aprobación para el git diff cada vez»
El shell con ! necesita esa tool pre-aprobada en allowed-tools.
«Claude lanza solo un comando con efectos»
Ponle disable-model-invocation: true.
«Un comando y el CLAUDE.md se contradicen»
Duplicaste. No repitas: el comando se refiere a las convenciones sin reenunciarlas.

Resumen de la lección

Modulariza: el CLAUDE.md raíz es el índice; lo profundo o específico de una zona va en reglas con ámbito (@ o .claude/rules/).

Una regla es contexto: se escribe como el CLAUDE.md —imperativo, concreto, énfasis con moderación.

Un slash command es un fichero markdown = un comando. Frontmatter, argumentos ($ARGUMENTS, $0) y shell embebido con !.

allowed-tools restringe por seguridad; disable-model-invocation reserva los comandos con efectos para cuando los tecleas tú.

La frontera: saber siempre (CLAUDE.md), saber según dónde (reglas), hacer cuando lo pides (comandos).
