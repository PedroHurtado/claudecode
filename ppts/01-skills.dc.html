Sesión 3 · Skills, agentes, MCPs y hooks
Dónde estamos

Cuando un comando se queda corto, la señal es: toca una skill

Necesitas que Claude lo invoque solo cuando es relevante, o que arrastre sus propios ficheros de apoyo. Hoy abrimos esa caja.

El concepto

Qué es una skill

Una carpeta con un fichero SKILL.md dentro, más —opcionalmente— scripts, referencias y plantillas. Claude la carga cuando la juzga relevante.

Es el manual de onboarding que le escribirías a un compañero nuevo para una tarea concreta.

«Así desplegamos nosotros». Conocimiento procedimental, específico, reutilizable.

El punto que separa una skill útil de una inútil

Una skill no es documentación de una API que ya existe

La pregunta de control al escribir una skill: «¿esto se lo enseño a Claude porque no lo sabe, o lo copio de un sitio donde ya está?» Si es lo segundo, bórralo. Lo que aporta una skill es lo que el modelo no tiene de serie: cómo hacéis las cosas vosotros.

La diferencia, en una frase

Skill vs. slash command

/comando

Un prompt reutilizable que tú disparas tecleándolo.

Skill = eso + dos cosas
  • Claude la invoca solo cuando la detecta relevante
  • Puede arrastrar ficheros: scripts, referencias, plantillas

Se fusionaron: la skill es el formato recomendado; el comando, su versión mínima de un fichero.

El criterio

Cuándo crear una skill (y cuándo no)

✓Créala cuando
  • Repites un procedimiento especializado y multi-paso
  • Se beneficia de ficheros de apoyo: script, plantilla, referencia larga
  • Quieres que Claude lo aplique solo, sin pedírselo
✕No la crees cuando
  • Es una instrucción de una línea que aplica siempre → CLAUDE.md
  • Claude ya lo sabe hacer (se «infra-disparan» en lo trivial)
  • Es documentación de algo ya documentado fuera
La frontera clara

CLAUDE.md vs. skills

CLAUDE.md

Lo que Claude debe saber siempre y que es corto. «Usa async/await», «tests con make test».

Skill

Conocimiento especializado que solo importa en esa tarea y que puede ser extenso. Disponible sin ensuciar el contexto base: solo entra cuando hace falta.

→ Eso es progressive disclosure.

Por dentro

La estructura de una skill

mi-skill/
├── SKILL.md      (obligatorio)
│   ├── frontmatter name + description
│   └── instrucciones (markdown)
├── scripts/      código determinista
├── references/   docs a demanda
└── assets/       material de salida

El nombre de la carpeta es el identificador de la skill.

.claude/skills/ (proyecto, en git) o ~/.claude/skills/ (personal).

Si una de proyecto y una personal comparten nombre, gana la de proyecto.

Por qué escala sin penalizar el contexto

Progressive disclosure: carga en tres niveles

Nivel 1 · al arrancar
El frontmatter

name + description. Claude solo sabe que existe y cuándo usarla. Por eso puedes tener muchas.

Nivel 2 · si es relevante
El cuerpo del SKILL.md

Las instrucciones. Claude las lee solo si decide que la skill aplica, tras leer la descripción.

Nivel 3 · cuando hace falta
Los ficheros de apoyo

Un references/ de 2.000 líneas no gasta ni un token hasta que Claude lo abre.

Consecuencia práctica: mantén el SKILL.md corto (< 500 líneas) y mueve el detalle a references/.

Lo que de verdad decide si funciona

El campo description

Dos errores fatales

Vago: «Ayuda con informes». Claude no sabe cuándo dispararla.

Tímido: tiende a infra-dispararse: no la usa cuando sería útil.

El antídoto: descripciones «insistentes»

Di qué hace y cuándo usarla, enumerando los disparadores con las palabras que aparecerían en la petición. En tercera persona y enfocada en el cuándo.

«Úsala siempre que el usuario mencione incidencias, postmortems o un fallo de producción, aunque no diga "informe".»

Un buen ejemplo

El frontmatter enumera los disparadores

---
name: incident-report
description: >
  Genera informes de incidencia con el formato y las
  secciones que usa el equipo. Úsala cuando el usuario
  mencione una incidencia, un postmortem, un fallo de
  producción, o pida documentar un corte de servicio,
  aunque no diga la palabra "informe".
---
Saber cuál usar es criterio puro

Scripts, referencias y plantillas

scripts/
Para lo determinista

Fiabilidad exacta o código que Claude reescribiría. Eficiente en tokens y hace lo mismo siempre.

references/
Conocimiento a demanda

Esquemas de BD, plantillas, especificaciones. Claude las abre solo si la tarea lo pide.

assets/
Lo que va en la salida

Plantillas de documento, logos, fuentes, iconos. Material que acaba en el resultado.

Las tres, juntas

Un SKILL.md que reparte el trabajo

# Notas de versión

## Procedimiento
1. Ejecuta scripts/collect_commits.sh
   para reunir commits desde el último tag.
2. Agrupa por tipo según
   references/categorias.md.
3. Redacta con assets/plantilla-release.md.
4. Resalta los breaking changes al principio.
  • El qué y el cómo: en el SKILL.md, corto.
  • Recoger commits: en un script, determinista.
  • Categorías y plantilla: en ficheros que se cargan solo al usar la skill.
En la práctica

Crear una skill: dos caminos

A mano

Carpeta, SKILL.md con frontmatter, ficheros de apoyo. Empieza simple (markdown, sin scripts) y añade complejidad cuando ya funcione. Prueba tras cada cambio.

Con ayuda de Claude

Le describes lo que quieres y te genera la estructura: la carpeta, el SKILL.md formateado y los recursos. El camino rápido para el primer borrador; luego lo afinas.

Cómo redactar el SKILL.md

Una skill se escribe para que la use otra instancia de Claude

Lo va a leer otro Claude que no tiene tu contexto. Incluye lo útil y no obvio —el conocimiento procedimental, los detalles del dominio, los recursos—. No incluyas lo que cualquier Claude ya sabría hacer.

Pueden traer y ejecutar código

Precauciones

El frontmatter entra en tu system prompt

Una skill no confiable es vía de inyección. Trátalas como cualquier dependencia: con revisión.

Cuidado con los scripts

Un script se ejecuta. Revisa qué hace antes de confiar en una skill ajena que trae código.

Composición sin control explícito

Claude puede combinar varias skills en una tarea. Mantén tu conjunto acotado y entendible.

Datos sensibles

No metas credenciales ni datos personales, sobre todo si la versionas. Como en el CLAUDE.md.

Para practicar

Convierte un comando en skill

1Crea .claude/skills/<nombre>/ con su SKILL.md.
2Escribe una description con disparadores explícitos, en tercera persona e «insistente».
3Mueve un detalle a references/ y, si encaja, un paso determinista a scripts/.
4Comprueba que Claude la invoca sin que la teclees, con una tarea que encaje.
Para llevarse a casa

Resumen de la lección

Una skill es una carpeta con un SKILL.md que Claude invoca solo y que arrastra scripts, referencias y plantillas.

No es documentar lo ya documentado: encapsula tu conocimiento de proyecto, no la API de Anthropic.

Progressive disclosure carga en tres niveles → mantén el SKILL.md corto y mueve el detalle a references/.

El description —tercera persona, «insistente», con disparadores— decide si funciona.

Pueden ejecutar código: trátalas como dependencias y revísalas.
